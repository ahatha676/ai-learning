# 🐍 02 — 异常处理 (try/except)

## 为什么需要异常处理？

网络请求可能失败、文件可能不存在、JSON 可能格式不对。如果不处理，程序会**直接崩溃**。

异常处理让程序在出错时能**优雅地继续运行**，并返回有意义的错误信息。

## 代码示例

```python
try:
    response = requests.get(url)          # 可能出错
    response.raise_for_status()           # 如果 HTTP 状态码不是 200，抛出异常
    data = response.json()                # 可能 JSON 解析失败
    return data['current_condition'][0]   # 可能 KeyError 或 IndexError

except requests.exceptions.RequestException as e:
    # 网络层面的错误（DNS 失败、超时、连接拒绝等）
    return f"错误:查询天气时遇到网络问题 - {e}"

except (KeyError, IndexError) as e:
    # 数据解析层面的错误（API 返回格式变了）
    return f"错误:解析天气数据失败，可能是城市名称无效 - {e}"
```

## 异常处理结构

```
try:
    ← 放"可能出错的代码"
except 异常类型 as 变量:
    ← 放"出错后怎么办"
else:              ← （可选）没出错时执行
    ...
finally:          ← （可选）不管有没有错都执行
    ...
```

## 常见异常类型

| 异常 | 什么时候触发 |
|------|------------|
| `KeyError` | 字典里没有这个键，如 `data['aaa']` |
| `IndexError` | 列表索引越界，如 `lst[100]` |
| `ValueError` | 值不合法，如 `int("abc")` |
| `TypeError` | 类型不匹配，如 `"1" + 1` |
| `FileNotFoundError` | 文件不存在 |
| `requests.exceptions.RequestException` | requests 库的所有网络错误 |
| `Exception` | **所有异常的基类**（通常不建议裸用，太宽泛） |

## 最佳实践

```python
# ❌ 不好：捕捉了所有异常，但不知道到底出了什么问题
try:
    ...
except Exception as e:
    return f"出错了: {e}"

# ✅ 好：先捕捉具体的异常，最后再兜底
try:
    ...
except KeyError as e:
    return f"缺少字段: {e}"
except Exception as e:
    return f"未知错误: {e}"
```

# 1. 网络错误 → requests.exceptions.RequestException

# tools/get_attraction.py 中用了兜底

# agent_core/llm_client.py 中 LLM 调用也加了异常保护
